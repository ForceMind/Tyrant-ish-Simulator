# UI 设计稿与评审流程

## 先确认再开发

后续新增页面或大改页面，先走这个流程：

1. 先出生图位图效果图和交互说明。
2. 确认信息层级、按钮入口、PC/移动端布局。
3. 再进入代码实现。
4. 实现后用浏览器截图和 PNG 效果稿对照验收。

## 官网首页信息层级

目标：先让新玩家知道这是什么游戏，再给两个清晰入口。

- 第一层：游戏名称、类型、核心卖点。
- 第二层：主按钮 `进入游戏大厅`，副按钮 `快速体验`、`史官档案`。
- 第三层：视觉化王朝卡片，展示“选择会带来数值和暗流后果”。
- 第四层：三张卖点卡：完整模式、快速版、黑色幽默。

## 快速版设计原则

快速版不是完整模式的简化菜单，而是一条独立体验线：

- 开局像人生模拟器：先选底色、习惯、麻烦、速度。
- 过程像压缩王朝传记：每次只给关键节点。
- 玩家仍有参与感：保留 5 到 10 个关键选择。
- 结局必须快：几分钟内看到荒唐但合理的史官评价。

## PC / 移动端隔离原则

- PC 端优先横向信息密度，避免横向滚动条交互。
- 移动端优先单手点击和纵向阅读，卡片允许分页，但按钮必须明显。
- 移动端底部数值条固定紧凑展示，不和正文区混排。
- PC 和移动端样式必须通过断点隔离，不能互相污染。

## 已落地实现记录

- 官网首屏已经按 PNG 效果图方向实现：左侧品牌和入口，右侧王朝预览卡，底部卖点卡。
- 官网页面不再展示“无后端 / localStorage 存档 / 手机竖屏优先”等开发提示，全部替换为玩家可见的卖点。
- 快速版参数页在移动端采用真实分页，而不是横向滚动隐藏内容；每页只渲染当前参数组，避免屏外按钮。
- 快速版推演页采用事件卡、红色皇帝拍板区、速写年表、底部数值条的结构，选项与正文有明显视觉区分。
- PC 端官网和快速版保持分栏布局，移动端规则单独写在 `responsive.css`，避免互相影响。

## 待确认点

- 官网是否需要更强的品牌视觉图，还是保持轻量 H5 小游戏风格。
- 快速版是否需要写入史官档案，目前设计为不写入，避免污染完整模式图鉴。
- 快速版关键选择数量当前由推演速度控制，后续可改成玩家自选。
